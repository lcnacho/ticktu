## Successes & Strengths

1. **Infrastructure-first architecture paid dividends** — proxy.ts with subdomain routing, belt-and-suspenders tenant isolation (RLS + app-level tenantId), and foundational utilities (AppError, usePolling, dates.ts) established in Epic 1 held across all 9 subsequent epics with zero rework.

2. **100% FR coverage with zero gaps** — 39 functional requirements mapped and delivered. The requirements inventory and FR coverage map prevented the classic "we forgot this feature" problem.

3. **Implementation velocity via layered approach** — schemas first, then actions, then UI. The layered approach (all schemas → all actions → all routes → all UI) reduced integration issues because each layer was internally consistent.

4. **Adversarial code review caught 3 critical bugs** — Race condition on ticket capacity (Story 4.2), missing event_access_codes table (Story 7.1), and no webhook reconciliation cron (Story 4.3). Without the review cycle, these would have been production-breaking.

5. **Surgical descoping on Epic 7** — Offline scanning descoped to online-first. Every affected artifact (FR-VA-04, FR-VA-05, NFR-AV-03, NFR-DI-01, NFR-DI-02, and 2 UX-DRs) updated with strikethrough + replacement + date. Full traceability preserved.

6. **Consistent patterns across codebase** — 5-step Server Action pipeline (auth → authorize → validate → execute → after), tenantId as first param on every query, snake_case DB ↔ camelCase TS via Drizzle, integer cents for money, no external date/number libraries.

## Challenges & Growth Areas

1. **Commit granularity too coarse** — Epics 3-10 backend shipped in a single 62-file commit. Makes git bisect difficult and increases risk per deployment. Team agreed: one commit per story minimum going forward.

2. **Test coverage gap** — 21 test files for 149 source files (~14% by file count). Critical paths (purchase flow, webhook reconciliation, refund processing, concurrent capacity) lack integration tests. No E2E test suite exists.

3. **Stories 1.5 and 8.2 needed post-hoc fixes** — Stories marked "done" required fixes in the final commit. Indicates definition of done wasn't tight enough during development.

4. **Critical bugs found in review, not in development** — The 3 critical issues (race condition, missing table, no reconciliation) should have been caught during story development, not post-hoc adversarial review. Need to push quality checks earlier.

5. **No-barrel-files constraint has DX cost** — 24 schema files with direct imports everywhere. Correct architectural decision, but the developer experience of managing import paths needs acknowledgment.

---

## Key Patterns & Insights

1. **Infrastructure-first pays off** — Foundational utilities (AppError, usePolling, dates.ts) assigned to Story 1.1 were reused across all subsequent epics. Shared infra should always be in the first story.

2. **Code review is the real quality gate** — 22 fixes across all severity levels. The system works, but at the wrong stage. Fix: push checks earlier with schema completeness validation and concurrency checklists.

3. **Schema-first creates consistency** — 24 schema files all following identical conventions made the action and query layers predictable. This pattern should continue.

4. **Descoping template established** — Epic 7's descoping process (update all FRs, NFRs, UX-DRs with strikethrough + replacement + date) is the template for any future descoping.
